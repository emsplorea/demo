⚠ Tunge koblinger — krever arkitekturendring

Disse berører kjernefunksjonalitet og krever redesign av modulgrenser.

1. Autentisering — BFF-kobling mot .NET OIDC breaking

Berørte filer: ~8 Estimat: 3–5 dager Risiko: Høy
▶

Hva er koblet

Hele auth-flyten er hardkodet mot et .NET BFF-mønster med spesifikke endepunkter:

  • /bff/session — sesjonssjekk
  • /bff/login?returnUrl=... — innlogging via OIDC redirect
  • /bff/logout — utlogging med cookie-sletting
  • window.__APP_CONFIG__.BACKEND_URI — runtime config peker til BFF
  • Cookie-håndtering (SameSite, Secure, HttpOnly) styrt av .NET backend

Foreslått løsning — Auth Adapter Pattern

Abstraher auth bak et interface som hver installasjon implementerer:

// src/lib/auth/auth-adapter.ts export interface AuthAdapter { checkSession(): Promise<AuthUser | null> login(returnUrl?: string): void logout(): Promise<void> getAccessToken?(): Promise<string | null> } // Implementasjoner: // src/lib/auth/adapters/bff-adapter.ts → RmsV/.NET BFF // src/lib/auth/adapters/plorea-adapter.ts → Plorea auth (f.eks. Supabase/Auth0) // src/lib/auth/adapters/amendo-adapter.ts → Amendo auth

Konkrete steg

  • Definer AuthAdapter interface med session/login/logout
  • Flytt all BFF-logikk til bff-adapter.ts (bevarer nåværende funksjon)
  • Lag en AuthProvider React context som tar adapter via config
  • Erstatt alle direkte /bff/*-kall med adapter-kall
  • Fjern hardkodet clearClientSession() — la adapteren styre cleanup

2. API-lag — Hardkodet mot RmsV-endepunkter breaking

Berørte filer: ~12 Estimat: 4–6 dager Risiko: Høy
▶

Hva er koblet

  • Direkte kall til /api/Items/GetItemsEAN for varekatalog
  • RmsV-spesifikke DTO-er: InventoryListDto, InventoryDto, ItemEAN
  • IndexedDB-skjema er modellert direkte på RmsV-datastruktur
  • Hooks (useItemEANs, useInventories) har inline fetch-logikk
  • Backend types i backend-types.ts speiler .NET DTO-er 1:1

Foreslått løsning — Repository Pattern med Domain Types

// src/domain/inventory.ts — agnostiske typer export interface InventoryItem { id: number itemCode: string itemLabel: string eanCodes: string[] quantities: PackageQuantities } // src/lib/api/api-adapter.ts export interface InventoryApiAdapter { getInventories(stockId: number): Promise<Inventory[]> getItemCatalog(): Promise<CatalogItem[]> submitCount(data: CountSubmission): Promise<void> startInventory(params: StartParams): Promise<Inventory> } // Implementasjoner mapper mellom domene og backend: // adapters/rmsv-api.ts → /api/Items/GetItemsEAN etc. // adapters/plorea-api.ts → Plorea REST/GraphQL // adapters/amendo-api.ts → Amendo endpoints

Konkrete steg

  • Definer domene-typer som er agnostiske for backend
  • Lag InventoryApiAdapter interface
  • Flytt eksisterende fetch-logikk til rmsv-api.ts adapter
  • Lag mapper-funksjoner: RmsvDto → DomainType
  • Oppdater hooks til å bruke adapter fra context
  • Abstraher IndexedDB-skjema til domene-typer

3. Oversettelser — Avhengighet til RmsVGlobalization breaking

Berørte filer: ~6 Estimat: 2–3 dager Risiko: Moderat
▶

Hva er koblet

  • generate-i18n.js leser .resx fra RmsVGlobalization-prosjektet
  • Avhengig av .NET prosjektstruktur og filsti til Stockcontrol*.resx
  • Build-prosessen (npm run i18n) feiler uten tilgang til .NET-prosjektet
  • Oversettelsesnøkler bruker StockcontrolTexts namespace

Foreslått løsning — Self-contained i18n

  • Flytt ferdiggenererte JSON-filer (public/locales/) inn i repoet som source of truth
  • Behold generate-i18n.js som valgfri dev-tool for RmsV-installasjon
  • Legg til en i18n.config.ts som definerer tilgjengelige språk per installasjon
  • Plorea kan legge til egne nøkler uten å berøre .resx-filer
  • Vurder å flytte til next-intl eller i18next med JSON-filer direkte

4. Deployment — IIS-spesifikk med /StockControl basePath refactor

Berørte filer: ~5 Estimat: 1–2 dager
▶

Hva er koblet

  • basePath: '/StockControl' hardkodet i next.config.js
  • getAssetPath() utility returnerer /StockControl/...
  • web.config med IIS URL Rewrite-regler
  • Hele deployment-guiden forutsetter Windows Server + IIS
  • PowerShell copy-kommandoer i dokumentasjon

Foreslått endring

  • Gjør basePath konfigurerbar via miljøvariabel: NEXT_PUBLIC_BASE_PATH
  • Behold web.config som valgfri for IIS-deployment
  • Legg til Dockerfile og docker-compose.yml for containerisert deployment
  • Legg til Vercel/Netlify config som alternativ
  • Dokumenter deployment per plattform: IIS, Docker, Vercel

◈ Foreslått gjennomføringsplan

Fire faser, der fase 1–2 gir umiddelbar verdi uten å knekke eksisterende RmsV-funksjonalitet.

Fase 1 — Abstraksjonslag Uke 1–2

Definer interfaces: AuthAdapter, InventoryApiAdapter, BrandConfig. Flytt eksisterende logikk til RmsV-adaptere. Null funksjonalitetsendring — bare omstrukturering. Etter dette kompilerer og kjører appen identisk som før.

Fase 2 — Self-contained i18n + Branding Uke 2–3

Frigjør oversettelser fra .resx-avhengighet. Implementer BrandConfig med theme/logo/navn. Lag build-script for PWA-ikoner og manifest per brand. Appen kan nå deployes uten tilgang til .NET-prosjektet.

Fase 3 — Plorea/Amendo-adaptere Uke 3–5

Implementer plorea-auth-adapter og plorea-api-adapter. Definer Plorea-spesifikke endepunkter og mapper dem til domene-typer. Test fullstendig inventarflyt mot Plorea backend. Gjenta for Amendo.

Fase 4 — Deployment-fleksibilitet Uke 5–6

Legg til Dockerfile, Docker Compose, og CI/CD-pipeline. Dokumenter deployment for alle plattformer. Flytt VS-integrasjon til valgfri mappe. Publiser som intern npm-pakke eller git submodule.
