⚠ Heavy couplings — require architectural changes

These affect core functionality and require redesigning module boundaries.

1. Authentication — BFF coupling to .NET OIDC breaking

Affected files: ~8 Estimate: 3–5 days Risk: High
▶

What is coupled

The entire auth flow is hardcoded against a .NET BFF pattern with specific endpoints:

  • /bff/session — session check
  • /bff/login?returnUrl=... — login via OIDC redirect
  • /bff/logout — logout with cookie clearing
  • window.__APP_CONFIG__.BACKEND_URI — runtime config points to BFF
  • Cookie handling (SameSite, Secure, HttpOnly) controlled by .NET backend

Proposed solution — Auth Adapter Pattern

Abstract auth behind an interface that each installation implements:

// src/lib/auth/auth-adapter.ts export interface AuthAdapter { checkSession(): Promise<AuthUser | null> login(returnUrl?: string): void logout(): Promise<void> getAccessToken?(): Promise<string | null> } // Implementations: // src/lib/auth/adapters/bff-adapter.ts → RmsV/.NET BFF // src/lib/auth/adapters/plorea-adapter.ts → Plorea auth (e.g. Supabase/Auth0) // src/lib/auth/adapters/amendo-adapter.ts → Amendo auth

Concrete steps

  • Define AuthAdapter interface with session/login/logout
  • Move all BFF logic to bff-adapter.ts (preserves current functionality)
  • Create an AuthProvider React context that receives the adapter via config
  • Replace all direct /bff/* calls with adapter calls
  • Remove hardcoded clearClientSession() — let the adapter manage cleanup

2. API layer — Hardcoded to RmsV endpoints breaking

Affected files: ~12 Estimate: 4–6 days Risk: High
▶

What is coupled

  • Direct calls to /api/Items/GetItemsEAN for item catalog
  • RmsV-specific DTOs: InventoryListDto, InventoryDto, ItemEAN
  • IndexedDB schema is modeled directly on RmsV data structures
  • Hooks (useItemEANs, useInventories) contain inline fetch logic
  • Backend types in backend-types.ts mirror .NET DTOs 1:1

Proposed solution — Repository Pattern with Domain Types

// src/domain/inventory.ts — backend-agnostic types export interface InventoryItem { id: number itemCode: string itemLabel: string eanCodes: string[] quantities: PackageQuantities } // src/lib/api/api-adapter.ts export interface InventoryApiAdapter { getInventories(stockId: number): Promise<Inventory[]> getItemCatalog(): Promise<CatalogItem[]> submitCount(data: CountSubmission): Promise<void> startInventory(params: StartParams): Promise<Inventory> } // Implementations map between domain and backend: // adapters/rmsv-api.ts → /api/Items/GetItemsEAN etc. // adapters/plorea-api.ts → Plorea REST/GraphQL // adapters/amendo-api.ts → Amendo endpoints

Concrete steps

  • Define domain types that are backend-agnostic
  • Create InventoryApiAdapter interface
  • Move existing fetch logic to rmsv-api.ts adapter
  • Create mapper functions: RmsvDto → DomainType
  • Update hooks to consume the adapter from context
  • Abstract IndexedDB schema to domain types

3. Translations — Dependency on RmsVGlobalization breaking

Affected files: ~6 Estimate: 2–3 days Risk: Moderate
▶

What is coupled

  • generate-i18n.js reads .resx files from the RmsVGlobalization project
  • Depends on .NET project structure and file paths to Stockcontrol*.resx
  • Build process (npm run i18n) fails without access to the .NET project
  • Translation keys use the StockcontrolTexts namespace

Proposed solution — Self-contained i18n

  • Move pre-generated JSON files (public/locales/) into the repo as source of truth
  • Keep generate-i18n.js as an optional dev tool for RmsV installations
  • Add an i18n.config.ts that defines available languages per installation
  • Plorea can add its own keys without touching .resx files
  • Consider migrating to next-intl or i18next with JSON files directly

⚡ Moderate couplings — require refactoring

These can be resolved without fundamentally changing the architecture.

4. Deployment — IIS-specific with /StockControl basePath refactor

Affected files: ~5 Estimate: 1–2 days
▶

What is coupled

  • basePath: '/StockControl' hardcoded in next.config.js
  • getAssetPath() utility returns /StockControl/...
  • web.config with IIS URL Rewrite rules
  • Entire deployment guide assumes Windows Server + IIS
  • PowerShell copy commands in documentation

Proposed changes

  • Make basePath configurable via environment variable: NEXT_PUBLIC_BASE_PATH
  • Keep web.config as optional for IIS deployments
  • Add Dockerfile and docker-compose.yml for containerized deployment
  • Add Vercel/Netlify config as alternatives
  • Document deployment per platform: IIS, Docker, Vercel

◈ Proposed implementation plan

Four phases, where phases 1–2 deliver immediate value without breaking existing RmsV functionality.

Phase 1 — Abstraction layer Week 1–2

Define interfaces: AuthAdapter, InventoryApiAdapter, BrandConfig. Move existing logic into RmsV adapters. Zero functionality change — pure restructuring. After this, the app compiles and runs identically to before.

Phase 2 — Self-contained i18n + Branding Week 2–3

Free translations from .resx dependency. Implement BrandConfig with theme/logo/name. Create build script for PWA icons and manifest per brand. The app can now be deployed without access to the .NET project.

Phase 3 — Plorea/Amendo adapters Week 3–5

Implement plorea-auth-adapter and plorea-api-adapter. Define Plorea-specific endpoints and map them to domain types. Test the complete inventory flow against Plorea backend. Repeat for Amendo.

Phase 4 — Deployment flexibility Week 5–6

Add Dockerfile, Docker Compose, and CI/CD pipeline. Document deployment for all platforms. Move VS integration to an optional folder. Publish as an internal npm package or git submodule.
