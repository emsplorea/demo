JWT Claims Structure
{ "sub": "usr_8f3k", // user id "org": "comp_12", // company id "loc": "loc_47", // location id "aud": "plorea", // audience (fixed) "exp": 1710000300, // 5 min from iat "iat": 1710000000, "jti": "a1b2c3d4", // unique, anti-replay "env": "prod", // reject if mismatch "role": "manager", "surfaces": ["favorite-setup", "device-page", "ops-console", "order-overview"], "surfacesVersion": 1, // contract version "target": "device-page", // deep link destination "returnUrl": "https://rmsv.example.com/locations/47" }
Required Headers
Authorization: Bearer {jwt} X-Plorea-Env: prod Content-Type: application/json
Validation Steps
Fetch JWKS from /.well-known/jwks.json
Verify RS256 signature · check exp + aud
Reject if env ≠ current environment
Reject replayed jti (dedup window: 10 min)
External roles NEVER trusted directly — mapped via role matrix
Session Model
JWT used only for bootstrap — consumed on first use
Internal session cookie: httpOnly, Secure, SameSite=Strict
Session TTL: 30 min idle, 8 hour absolute max
No refresh token — re-auth via new direct link
Claim → DB Resolution
org (comp_12) → companies.id loc (loc_47) → companies_addresses.id sub (usr_8f3k) → companies_users.id → users.id role (manager) → iam_groups.id (via role matrix mapping)

SymptomAdmin sees "access denied" after SSO redirect CauseExternal role (e.g. "super_admin") not found in Plorea role matrix ActionAdd mapping in role matrix or fix role claim in 3rd party OwnerShared (3rd party sends role, Plorea maps it)

Determinism Guarantee

Input Matrix
Synced catalog (menus_items, menus_categories, menus_groups)
Location schedule (operating hours)
Channel rules (QR/Online/Kiosk/Catering)

Lead time configuration
Override Precedence (highest → lowest)
1. Ops override (busy mode → block all) 2. Manual toggle (per-item/category disable) 3. Schedule rule (operating hours) 4. Channel rule (channel-specific limits) 5. Default (catalog = available)
Reason Codes
"available" → menus_item orderable "ops_override" → blocked by ops toggle "outside_hours" → location closed "channel_excluded" → not on this channel "lead_time_exceeded" → prep time too long

Order Core
All surfaces → same APIs · no surface-specific backends
› details
State Machine
Draft → PendingPayment → Paid → Confirmed → Pushed → Acknowledged │ │ │ └── (cancel) ────────┘── (payment fail → retry) ────┘── (webhook timeout → DLQ)
Allowed Transitions
Draft → PendingPayment (order submitted)
PendingPayment → Paid (Adyen callback)
Paid → Confirmed (auto, after payment verified)
Confirmed → Pushed (webhook dispatched)
Pushed → Acknowledged (3rd party 200 OK)
PendingPayment → Draft (payment fail, retry)
Any → Cancelled (manual or timeout)
Idempotency Rules
Idempotency-Key required on POST /v1/orders
Duplicate key → return existing order (not error)
Payment ref idempotent — Adyen handles dedup
Webhook EventId idempotent — 3rd party must dedup
Example 409 Conflict
POST /v1/orders/{id}/transition Body: {"to": "Paid"} 409 Conflict { "error": "invalid_transition", "currentState": "Draft", "requestedState": "Paid", "allowedTransitions": ["PendingPayment", "Cancelled"] }
Surface API Endpoints
GET /v1/availability/{locationId} Availability (computed) PUT /v1/availability/{locationId}/busy Ops busy-mode toggle GET /v1/orders?loc={locationId} Order history GET /v1/devices?loc={locationId} Device list POST /v1/favorites/{locationId} Save favorites
Governance Rules
Architecture Governance
No duplicated logic — availability computed once, used everywhere
No UI business logic — surfaces are presentation only
Extend don't fork — new channels reuse Order Core, never clone
Integration boundary — gate enforced, financial surfaces blocked
Deterministic availability — same input → same output, always
Role matrix mapping — external roles never trusted 1:1
Audit everything — every access logged with full context
No POS bypass — all orders go through Order Core state machine
🚫 What We Refuse To Do
✕ No financial reporting in Plorea (integration mode)
✕ No channel-specific order logic — unified engine only
✕ No UI-based business logic — surfaces are presentation layer
✕ No POS bypass — all orders through state machine
✕ No direct role passthrough — always mapped via matrix
