Executive Summary

Objective

Deliver a stable, simple and production-ready V1 of the Plorea Order Platform covering Settings, Frontend functionality and required integrations with RMSV and Amendo.

Core Principle

One Order Core. No duplicated business logic. Availability is computed. Operations override configuration. No channel-specific engines.

Integration Principle (Critical)

When RMSV or Amendo integration is active, Plorea acts strictly as the Order Platform. Financial reporting, accounting, end-of-day settlement and statistics remain fully owned by the 3rd-party system.

Architecture Boundary

Master data ownership remains in the 3rd-party system. Channel configuration and runtime control remain in Plorea. Partner frontends integrate with Plorea via a dedicated GraphQL API — Plorea does not host configuration UIs for partner users in V1, and no REST endpoints are exposed to partners.

Communication Method

  • Partner → Plorea (queries & mutations): GraphQL over POST /graphql/{partner}, authenticated via Sanctum bearer token
  • Plorea → Partner (order events & lifecycle push): WebSocket (real-time) or Webhook (event-driven HTTP callback)

GraphQL foundation work is tracked in PLBR-1152 (Amendo endpoint, login mutation, companies query, updateCompany mutation). Additional partner types and mutations are added incrementally per integration.

Governance Rule

No parallel reporting logic may be introduced in Plorea when 3rd-party integration is active. All financial ownership remains in RMSV or Amendo.

Partner Integration Model

When RMSV or Amendo integration is active, partner systems integrate with Plorea through partner-native frontends calling a dedicated Plorea GraphQL API. Configuration and operational surfaces are built and owned inside the partner system's own UI, consuming Plorea GraphQL queries and mutations.

Plorea does not host Direct Link / SSO configuration pages for partner users in V1. Plorea-hosted surfaces are used by Plorea-native customers and during internal operations, not embedded into partner workflows.

GraphQL Foundation · PLBR-1152

Each partner gets a dedicated GraphQL endpoint (e.g. POST /graphql/amendo) with its own schema namespace under app/GraphQL/{Partner}/. Authentication uses Sanctum personal access tokens via Authorization: Bearer <token>, with short-lived tokens (30 min for Amendo). The existing Plorea POS /graphql endpoint is unaffected. No REST endpoints are exposed to partners.

First endpoints (Amendo · PLBR-1152)

  • login(email, password): LoginPayload — mutation, returns token + expiry + user
  • companies(id, limit, offset): [CompanyType] — query, scoped to authenticated user
  • updateCompany(id, input: CompanyInput): CompanyType — mutation, authorization enforced per company

Surfaces Owned by Plorea (Plorea-native customers)

Favorite Setup

Catalog visibility per device and time window. Product master data originates from 3rd-party when integration is active; visibility configuration is managed in Plorea.

Device Page

Channel configuration (QR / Online / Kiosk / Catering).

Ops Console

Runtime operational control: temp close, busy mode, product disable, preorder approval. Ops overrides exist only in Plorea and override configuration.

Order Overview

Unified order inbox for operational visibility. Plorea provides operational visibility; 3rd-party system owns financial reporting. Does not replace accounting or settlement reporting.

Governance – Partner Integration
  • Partner API access authenticates via Sanctum bearer tokens with short-lived expiry
  • Each partner has an isolated GraphQL schema namespace; no shared mutations across partners
  • Must respect role permissions received from 3rd-party
  • Authorization checks enforced at resolver level (e.g. user must belong to the company being mutated)
  • Must not introduce financial reporting logic
  • Must not duplicate master data ownership
  • No REST endpoints exposed to partners
