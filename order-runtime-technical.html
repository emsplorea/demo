⚡ Simulate Failure — Order + Payment
PaymentFailed
SymptomCustomer sees "payment failed" on checkout CauseUser cancelled (Vipps abort), invalid card, insufficient funds ActionPayment state → Failed · order remains in PendingPayment · customer can retry with different method OwnerShared — customer action + Adyen validation
PaymentError → Retry
SymptomCustomer sees "try again" message CauseAdyen endpoint returns 5xx — transient failure ActionAuto-retry 3× with backoff · after max: PaymentError state · ops review → manual cancel or retry OwnerPlorea (retry logic) + Adyen (infrastructure)
Retry Initiated
SymptomOrder stays in Pushed state — not Acknowledged Cause3rd party webhook exceeds 5s timeout ActionRetry 5× (1s, 5s, 30s, 5m, 30m) · after max: DLQ + ops alert Owner3rd Party (must respond within 5s)
🔒
Payment Adapter
POST /v1/orders/{id}/pay · External Sync · Adyen
› details
Error Path (corrected)
200 → Authorized/Captured → Order moves to Confirmed 4xx → PaymentFailed (user cancelled / invalid method / Vipps abort) Order stays PendingPayment · customer can retry 5xx → Retry 3× w/ backoff → PaymentError (manual review or cancel) ⚠ 4xx is NOT a revert to Draft — it's a payment-level failure Order lifecycle and payment lifecycle are separate tracks
Idempotency
Payment ref idempotent — Adyen handles dedup on their side
Same payment ref → same result (no double charge)

Order Runtime — Governance

Architecture rules · ownership boundaries · what we refuse to do

Architecture Rules
Runtime Governance
Order + Payment = separate lifecycles — never conflate "Paid" with order status
Idempotency everywhere — Idempotency-Key on orders, eventId on webhooks, payment ref on Adyen
Integration Gate at runtime — checked before every order, not just at setup
Deterministic availability — same input → same output, always
At-least-once delivery — receiver must dedup on eventId
DLQ for webhook failures — never silently drop orders
4xx ≠ revert to Draft — payment failure is PaymentFailed, not order cancellation
locationId = companies_addresses.id — consistent across all endpoints
🚫 What We Refuse To Do
✕ No financial reporting in Plorea (integration mode)
✕ No channel-specific order logic — unified engine only
✕ No "order status = paid" conflation — separate lifecycles
✕ No settlement or Z-rapport — stays in 3rd party
✕ No silent webhook drops — DLQ + alert always
