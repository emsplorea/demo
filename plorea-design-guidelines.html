Plorea Design System

The standardized frontend platform used across all Plorea products — POS, kiosk, admin, KDS, online ordering, and QR surfaces.

Status: Active Blueprint v1.4 Owner: Einar Morten Slapgård Scope: Entire Plorea Frontend
Tailwind CSS daisyUI @plorea/components
01 — Overview

Purpose

The Plorea Design System is the standardized frontend platform used across all Plorea products. It defines the visual language, interaction patterns, and UI architecture used on every surface the platform ships.

Plorea does not attempt to reinvent low-level UI primitives. Instead it builds a curated, opinionated system on top of selected open-source foundations — keeping the investment focused on domain-specific UX rather than commodity components.

Goals
  • →Consistent user experience across all surfaces
  • →Rapid feature development without UI debt
  • →Performance on POS terminal hardware
  • →Scalability across markets and white-label partners
Applies to
  • ⬡POS Terminal
  • ⬡Admin Backoffice
  • ⬡Kitchen Display System (KDS)
  • ⬡Online Ordering / QR / Self-Service
Demo code vs production code Everything rendered in this guide — buttons, tables, badges, live demos — is documentation-only markup. Its classes (.card, .badge, .d-row…) exist to illustrate the system and must never be copied into the product. Rendered illustrations are marked Doc demo; copyable patterns sit in dark code blocks marked Production. Product code imports from @plorea/components — nothing else. This page practices what it preaches: every colour resolves from the token layer, stacking uses the named z-scale, spacing sits on the 4 px grid, focus is visible, and coarse-pointer targets meet 44 px.

07 — Policy

Vendor Usage Policy

To avoid UI fragmentation, vendor components must follow strict usage rules. Plorea components must remain the single UI contract across all product surfaces.

✓ Allowed
  • Using vendor components via Plorea wrapper components
  • Direct usage in prototype branches or experiments
  • Direct usage when creating a new canonical Plorea component
  • Using daisyUI classes inside @plorea/components source
✗ Not Allowed
  • Random badge-success classes scattered in product screens
  • Multiple visual implementations of the same UI pattern
  • App-specific theme definitions that override Plorea tokens
  • Importing directly from daisyUI in product application code
Principle If you find yourself writing class="badge badge-success" in a product screen, you should be writing <PStatusBadge status="paid" /> instead.

Vendor Update Policy

Third-party UI libraries such as daisyUI are treated as framework foundations — they must never be modified directly inside product code.

  • Vendor components must not be edited directly — wrap them instead.
  • Custom behaviour or styling must be implemented through Plorea wrapper components.
  • Applications must import UI only from @plorea/components — never directly from daisyUI.
  • Upgrades to vendor libraries must be validated against the full Plorea wrapper layer before merging to main.

This policy prevents UI fragmentation and ensures that vendor upgrades do not silently break existing Plorea surfaces.

09 — UX

POS UX Principles

Plorea POS interfaces follow strict UX principles designed for high-frequency, high-pressure operational environments. These rules are non-negotiable for POS and KDS surfaces.

01
Maximum two clicks for common actions
Any action a cashier performs more than 20 times per shift must be reachable in ≤ 2 taps from the main screen.
02
Primary actions always visible
The primary action on every screen is always above the fold and visually prominent. Never hidden behind a menu.
03
No hidden navigation layers
POS interfaces must not require discovery. All navigation paths are visible. No hamburger menus on operational screens.
04
Visual feedback under 100ms
All interactive elements provide visual feedback within 100ms. Use CSS transitions, not JS animations, for this layer.
05
Color communicates state
State colours are defined in tokens and are never overridden per-component. Green = ready, yellow = in progress, red = error/alert — always.
06
Avoid blocking modal flows
Modals that block the entire POS screen are only acceptable for payment flows and destructive confirmations. Never for informational content.
07
Interface must work without training
A new cashier with restaurant industry experience should be able to process a simple order within 5 minutes on their first day.
09b — UX

POS Environments — Terminal, Kiosk & Low Light

The seven UX principles apply everywhere, but the physical environment changes the numbers. A staff terminal is operated by trained hands at speed; a kiosk by a first-time guest standing up; an evening service floor is dim. Terminal — staff POS
  • Wet and greasy fingers are the norm: no precision gestures. Nothing may require long-press, drag or double-tap as the only path.
  • Repeat-tap protection: primary actions lock for 600 ms after tap (loading state) so a nervous double-tap can't fire two orders.
  • Destructive actions confirm through PModal with object + consequence — never rely on undo alone.
  • Density compact by default; ultra only on KDS-adjacent lists.
  • Everything a cashier does > 20×/shift: ≤ 2 taps from main screen (Principle 01).
Kiosk — guest self-service
  • First-time users, standing up: targets ≥ 64 px, body ≥ 18 px, action labels ≥ 24 px.
  • Vertical totems: keep all interaction inside the middle band — nothing critical in the top 15 % or bottom 10 % of the screen (reach).
  • One primary action per screen, full width at the bottom. No competing choices.
  • Idle reset after 90 s with a visible 10 s countdown. Cart and guest data are cleared — nothing persists between guests.
  • Customer-facing voice rules apply: no internal terms, no abbreviations (see Voice by surface).
  • Payment is the only permitted full-screen blocking flow (Principle 06).
Low light — evening service
  • Default theme dark-dim — built for reduced eye strain. Never pure white surfaces after dark; a bright panel blinds a dim room.
  • Critical text and status: contrast ≥ 7:1 against dark tokens. Verify per component, not per palette.
  • Status is never colour alone — icon + label mandatory. Red and green converge in dim light.
  • Focus ring visibility must be re-verified on dark tokens for every interactive component.
  • Venues may schedule theme by opening hours; manual override is always one tap away.

10 — Performance

Performance Guidelines

POS terminals are dedicated hardware, often with limited CPU resources. The UI must remain responsive during peak lunch and dinner rushes, with multiple concurrent orders updating in real time.

Render Rules
  • ✗Avoid heavy CSS animations on POS screens
  • ✓Target frame render under 16ms (60fps)
  • ✗Avoid deep component nesting (> 8 levels)
  • ✗Limit unnecessary state updates and re-renders
  • ✓Use skeleton loaders, not spinners
  • ✓Optimistic updates for order state changes
Touch Targets
  • →Minimum: 44×44px (WCAG AA)
  • →POS terminal: 48×48px preferred
  • →Kiosk dine-choice buttons: 120px minimum
  • →Destructive actions: always require confirmation
Loading & bundle The app currently ships as a single ~2 MiB JavaScript chunk with every route eagerly loaded. Split the bundle so a POS terminal only parses what it needs.
Code splitting
  • ✓Lazy-load routes with React.lazy + Suspense
  • ✓Split heavy vendors into their own chunks via manualChunks (konva, framer-motion, pusher)
  • ✓Lazy-load single-use features (table-map canvas, kiosk animations)
  • ✗Don't raise the PWA precache limit to mask an oversized entry chunk
Budget
  • →Entry chunk target: < 1 MiB gzipped
  • →Keep the Workbox precache limit at its 2 MiB default
  • →Track bundle size in CI (e.g. rollup-plugin-visualizer)
11 — Accessibility

Accessibility

Accessibility is a first-class requirement, not an afterthought. daisyUI provides styling and semantic class names, but not accessibility behaviour — Plorea wrapper components are responsible for verified accessibility, including focus management, keyboard handling, ARIA attributes and WCAG AA validation.

Requirement All canonical components must pass WCAG AA before they can be promoted from experimental to stable in the component registry.
Requirements
  • ✓Full keyboard navigation
  • ✓Screen reader compatibility (ARIA labels)
  • ✓High contrast colour combinations (4.5:1 min)
  • ✓Visible focus states on all interactive elements
Accessibility Ownership

daisyUI may provide useful semantic classes and visual primitives, but Plorea wrapper components are responsible for verified accessibility behaviour, including focus management, keyboard handling, ARIA attributes and WCAG AA validation. daisyUI is a CSS-only Tailwind plugin with no JavaScript behaviour, so this is not inherited automatically — focus trapping, keyboard handling and screen-reader semantics (e.g. role="dialog", aria-modal, aria-labelledby and a focus trap on Modal) must be implemented and tested in each wrapper.

Promotion gate A component cannot move from experimental to stable until every item below is verified. daisyUI does not provide this — the Plorea wrapper owns it.
Per-component checklist
  • ✓Visible focus ring on every interactive element (focus-visible)
  • ✓Full keyboard path, logical tab order, Esc closes overlays
  • ✓aria-label on standalone and icon-only controls
  • ✓Contrast ≥ 4.5:1 text / 3:1 UI; never colour as the only signal
  • ✓Touch targets ≥ 44px (POS 48 / kiosk 120)
  • ✓Green axe story in Storybook (addon-a11y) before promotion
Dialogs & modals

Every dialog wrapper must set role="dialog", aria-modal="true" and aria-labelledby on its title, trap focus while open and restore focus to the trigger on close, close on Esc, and use an icon close button with an aria-label — never a bare ✕ glyph. Use PDialog/PModal as the reference implementation. The legacy common/Modal meets none of this and must not be used in new code.

Tooling Add eslint-plugin-jsx-a11y so missing alt text, label-less controls and invalid ARIA fail at lint time — it is not yet installed.
12 — Money & Numbers

Money & Numbers

Plorea is a point-of-sale system: prices, VAT, discounts and change must be exact to the øre. JavaScript floating-point math is not — 0.1 + 0.2 !== 0.3 and 1.1 * 3 === 3.3000000000000003 — so money is never computed with raw number arithmetic patched by .toFixed().

Rule All monetary math goes through a single decimal money utility (decimal.js-based). Never add, multiply or subtract prices as raw floats; never use .toFixed() as a rounding strategy.
Calculation
  • ✓One utility for add / subtract / × quantity / VAT / discount / rounding
  • ✓VAT from a gross amount: gross − gross / (1 + rate/100)
  • ✓Round once, at the end, to the location's decimal precision
  • ✗parseFloat(x.toFixed(2)) used as arithmetic
  • ✗Mixing raw number and decimal values mid-calculation
Display & rounding
  • →Currency prefix and decimal precision come from the location (kr)
  • →Norwegian default: comma decimals, space thousands (kr 1 234,50)
  • →Right-align amounts; use font-variant-numeric: tabular-nums
  • →One documented rounding mode (app default rounds halves up); cash/øre rounding only where configured
Inputs are untrusted Treat price and VAT fields from the backend as possibly missing or string-typed. The money utility coerces invalid or absent values to 0 rather than letting NaN reach a displayed total — a missing VAT rate yields 0 VAT, never NaN.

PButton

Every tappable action in the product. One primary per view; pair fill + outline.

Production
// PButton.types.ts type PButtonVariant = 'primary' | 'secondary' | 'ghost' | 'destructive' type PButtonSize = 'md' | 'lg' | 'xl' // min-height 44 / 52 / 64 px interface PButtonProps extends BaseProps, LoadableProps, DisableableProps { variant?: PButtonVariant // default 'primary' size?: PButtonSize // default 'md' · kiosk surfaces resolve to 'xl' iconLeft?: ReactNode iconRight?: ReactNode fullWidth?: boolean type?: 'button' | 'submit' // default 'button' onPress: () => void // click + Enter/Space children: string // outcome-led: "Send to kitchen", never "Submit" } <PButton variant="destructive" onPress={confirmRefund}>Refund payment</PButton>
  • loading keeps the button's width, swaps the label to the in-progress verb ("Sending…") and locks re-taps for 600 ms — terminal double-tap protection is built in, not opt-in.
  • destructive is never the only button — always paired with a cancel path.
  • Touch target never below the surface minimum even at size="md" (padding grows, glyph doesn't).

PInput

Single-line text entry. Label always visible — placeholder is never the label.

Production
// PInput.types.ts interface PInputProps extends BaseProps, DisableableProps { label: string value: string onValueChange: (value: string) => void type?: 'text' | 'email' | 'tel' | 'number' | 'password' inputMode?: 'text' | 'numeric' | 'decimal' | 'tel' | 'email' // drives the touch keyboard placeholder?: string hint?: string // one line under the field error?: string // what went wrong + what to do next required?: boolean maxLength?: number prefix?: ReactNode suffix?: ReactNode }
  • Max 4 inputs per group (Form rule) — more means steps or sub-groups, not a longer form.
  • error follows UX Writing: "Enter a table number", never "Required field". Setting it announces via aria-describedby + aria-invalid.
  • Numeric fields set inputMode so terminals get the number pad — amounts use 'decimal', table numbers 'numeric'.

PModal

The only dialog. Replaces legacy common/Modal — owns focus trap, role="dialog", aria-modal, Esc, focus return, z-modal and elevation 3.

Production
// PModal.types.ts type PModalSize = 'sm' | 'md' | 'lg' | 'fullscreen' // fullscreen: payment flows only interface PModalAction { label: string // outcome-led: "Delete order" onAction: () => void loading?: boolean disabled?: boolean } interface PModalProps extends BaseProps { open: boolean onClose: () => void // Esc + overlay, unless blocking title: string size?: PModalSize // default 'md' destructive?: boolean // red primary; title carries object + identifier blocking?: boolean // payment / destructive only — Principle 06 primaryAction: PModalAction secondaryAction?: PModalAction initialFocusRef?: RefObject<HTMLElement> children: ReactNode }
  • Destructive confirmations follow the copy template: object + identifier in the title, concrete consequence in the body.
  • blocking removes Esc/overlay close — allowed only for payment flows and destructive confirmations. Informational content never blocks.
  • New code importing common/Modal fails design lint immediately.

PTable

Data lists and order tables. Density-aware, skeleton-loading, empty states with reason + action.

Production
// PTable.types.ts interface PTableColumn<T> { key: string header: string width?: string align?: 'left' | 'right' // money & counts: right + tabular-nums sortable?: boolean render?: (row: T) => ReactNode } interface PTableProps<T> extends BaseProps, LoadableProps { columns: PTableColumn<T>[] data: T[] rowKey: (row: T) => string density?: Density // default from useSurfaceEnvironment selectedKey?: string onRowSelect?: (row: T) => void stickyHeader?: boolean sort?: { key: string; dir: 'asc' | 'desc' } onSortChange?: (sort: { key: string; dir: 'asc' | 'desc' }) => void emptyState: { message: string; actionLabel?: string; onAction?: () => void } }
  • loading renders skeleton rows mirroring the column shape — never a spinner, never a blank surface.
  • emptyState.message includes the reason: "No orders match this filter", not "No data".
  • Row height, gaps and cell padding come from the density tokens (--row-h, --row-gap, --cell-py).

PBadge

Status and order-channel chips. Earlier drafts call this PStatusBadge — PBadge is the canonical name.

Production
// PBadge.types.ts type PBadgeChannel = 'table' | 'takeaway' | 'eatin' | 'combo' | 'online' | 'kiosk' interface PBadgeProps extends BaseProps { tone?: Tone // system status — paid, pending, failed… channel?: PBadgeChannel // order channel — mutually exclusive with tone pip?: boolean // default true — colour is never alone children: string // ≤ 2 words } <PBadge tone="success">Paid</PBadge> <PBadge channel="takeaway">Takeaway</PBadge>
  • Channel colours come exclusively from the --channel-* tokens — never overridden per component (Principle 05).
  • Passing both tone and channel is a type error, not a runtime warning.

PToast

Transient feedback. Imperative API — no JSX placement per screen; one viewport-level outlet at z-toast.

Production
// PToast.types.ts interface PToastOptions { title: string // ≤ 6 words, outcome-led: "Order #4821 sent to kitchen" description?: string // ≤ 1 sentence tone?: Tone // default 'info' duration?: number // ms · success min 3000 · 0 = persists until dismissed action?: { label: string; onAction: () => void } } toast(options: PToastOptions): ToastHandle toast.success('Order #4821 sent to kitchen') toast.error('Printer offline — check power and retry', { duration: 0 })
  • Success toasts never auto-dismiss under 3 seconds (Do / Don't). Error toasts default to duration: 0 — they persist until acted on.
  • Toasts never carry the only path to an action; they may shortcut one that exists elsewhere.
  • Body text follows the error template: what went wrong + what to do next.

13 — Guidance

React Component Token Rules

The token architecture only delivers dark mode (and future theme support) if components consume var(--color-*) tokens exclusively. A single bg-gray-100 in a component bypasses the entire swap mechanism. These are the rules that make the system work end-to-end.

✓ Correct — semantic token
// Card component
<div
  className="rounded-lg p-6"
  style={{
    background: 'var(--color-surface)',
    border: '1px solid var(--color-border)',
    boxShadow: 'var(--elevation-1)'
  }}
>
✗ Wrong — primitive Tailwind class
// Bypasses token swap — stays white in dark mode
<div
  className="bg-white border border-gray-200"
  className="rounded-lg p-6 shadow-sm"
>

// Also wrong — dark: modifier couples component to mode
<div className="bg-white dark:bg-gray-900">

Tailwind classes that are safe to use

Not all Tailwind utility classes break token contract. Layout, spacing, radius and typography scale classes are fine — they carry no colour information.

✓ p-6 px-4 py-2 — spacing
✓ gap-3 space-y-2 — layout
✓ rounded-lg rounded-xl — radius
✓ text-xs text-sm text-md — type scale
✓ font-bold font-semibold — weight
✓ flex items-center justify-between
✓ w-full h-11 min-w-0 — sizing
✓ truncate overflow-hidden
✓ transition duration-fast — motion
✓ z-modal z-dropdown — named z-index

✗ bg-white bg-gray-50 — use var(--color-surface)

Voice & Tone

UX Writing defines the rules for interactive text. This section defines how Plorea sounds — in the product, on receipts, in notifications and in marketing. The voice never changes; the volume adapts to the surface.

Principle 01
Calm authority

We state, we don't shout. No exclamation marks, no hype, no urgency theatre. A till used mid-rush has to sound steady — confident without being loud.

Principle 02
Outcome over process

Say what's done, not what the system is doing. "Sent to kitchen", not "Your request is being processed". Staff care about the result, not the mechanics.

Principle 03
High signal, low noise

Every word earns its place. If it doesn't help someone act, cut it. Silence is fine — white space is allowed to do the talking.

Human, not robot

Write like you'd say it to a colleague behind the counter. Contractions are fine. Corporate jargon is not. Clear beats clever.

These words are banned from Plorea copy — in the product, in release notes, on the marketing site. They're vague, overused, or sound like every other SaaS company. Delete them on sight.

Empty verbs

The Plorea Design System standardizes how user interfaces are built across the entire Plorea platform.
