Plorea UI Framework

The standardized frontend platform used across all Plorea products — POS, kiosk, admin, KDS, online ordering, and QR surfaces.

Status: Active Blueprint v1.1 Owner: Einar Morten Slapgård Scope: Entire Plorea Frontend
Tailwind CSS shadcn/ui daisyUI @plorea/components
01 — Overview

Purpose

The Plorea UI Framework is the standardized frontend platform used across all Plorea products. It defines the visual language, interaction patterns, and UI architecture used on every surface the platform ships.

Plorea does not attempt to reinvent low-level UI primitives. Instead it builds a curated, opinionated system on top of selected open-source foundations — keeping the investment focused on domain-specific UX rather than commodity components.

Goals
  • →Consistent user experience across all surfaces
  • →Rapid feature development without UI debt
  • →Performance on POS terminal hardware
  • →Scalability across markets and white-label partners
Applies to
  • ⬡POS Terminal
  • ⬡Admin Backoffice
  • ⬡Kitchen Display System (KDS)
  • ⬡Online Ordering / QR / Self-Service
02 — Tech Stack

Framework Foundations

The framework is built on top of four complementary technologies. Each plays a specific role — none of them overlap, and none of them are optional.

Layer Technology Role
Framework React + TypeScript UI runtime. Strict mode, concurrent features.
Styling Tailwind CSS Low-level utilities. Layout, spacing, responsive breakpoints.
Component primitives shadcn/ui Accessible, composable React primitives. Complex interactions.
Utility components daisyUI Semantic utility classes. Badges, alerts, stats, rapid prototyping.
Bundler Vite Fast HMR, ESM-native, library build mode.
Data handling TanStack Query Server state, caching, optimistic updates.
Backend integration Laravel + Inertia SSR-friendly SPA. Shared data model with backend.
Documentation Storybook Visual reference and development environment.

Role of Each Foundation

Tailwind
  • Low-level utility styling
  • Layout and spacing
  • Responsive breakpoints
  • The cn() utility for class composition
shadcn/ui
  • Composable React primitives
  • Accessible interaction patterns (ARIA)
  • Complex UI: dialogs, dropdowns, command palettes
  • Foundation for Plorea wrapper components
daisyUI
  • Semantic utility classes
  • Lightweight visual components
  • Rapid UI composition in prototypes
  • Theme layer — maps to Plorea tokens via CSS vars
Plorea Framework
  • Design tokens (single source of truth)
  • Wrapper components with Plorea variants
  • Domain components (orders, payments, KDS)
  • UX rules and POS performance standards

04 — Styling

Styling Hierarchy

The framework defines a strict styling hierarchy. Following this order ensures consistent theming, dark mode support, and maintainability across the codebase.

1
Plorea Design Tokens
CSS custom properties. var(--color-primary), var(--r-lg), var(--dur-fast). Never use raw hex values in components.
2
Plorea Wrapper Components
Use PButton, PInput, PDialog etc. These components already apply the right tokens and variants.
3
shadcn/ui Primitives
Used inside Plorea wrappers. Not consumed directly in product code.
4
daisyUI Semantic Classes
Used inside Plorea wrappers for quick composition. Not used directly in product screens.
5
Tailwind Utilities
Layout, spacing, and responsive helpers. Used everywhere via className. Color classes only via token-mapped aliases.
6
Raw CSS — by exception only
Only when no other layer can achieve the result. Must be documented with a comment explaining why.
05 — Design Tokens

Design Tokens

Design tokens define the core visual language of Plorea. They are the single source of truth for all colour, spacing, radius, shadow, and motion values. Raw values must never be used in components.

Token Structure
tokens/ colors.ts ← surface, text, semantic spacing.ts ← scale + semantic typography.ts ← size, weight, family radius.ts ← --r-sm through --r-full shadows.ts ← elevation 0–4 motion.ts ← dur-fast, dur, ease-out index.ts ← barrel export
Example Token Definition
export const colors = { surface: "#FFFFFF", surfaceElevated: "#F7F8FA", border: "#E9E9E9", textPrimary: "#1A1A1A", textMuted: "#C4C4C4", primary: "#6BB8DA", success: "#6BDAB2", warning: "#DADA6B", danger: "#DA6B6B", }

06 — Components

Component Architecture

All components in @plorea/components are divided into three categories with distinct responsibilities. The category determines where a component lives, how it is tested, and who consumes it.

🧱
Foundation Components
Based on shadcn/ui primitives. Provide interaction logic and accessibility. Not consumed directly in product code.

These components form the interaction and accessibility layer. They are wrapped by Plorea components before exposure to product teams.

Button Input Dialog Dropdown Tabs Popover Sheet Command Form Drawer
🎨
Plorea Wrapped Components
Wrap foundation components and apply Plorea styling, variants, and token mapping. These are the public UI API for product teams.

Applications should use these components instead of direct vendor components. They carry Plorea's visual language and UX rules.

PButton PInput PSelect PDialog PBadge PCard PAlert PTabs PDropdown PToast
🏪
Domain Components
Represent business functionality. Encapsulate POS, order, and payment domain logic. These do not exist in shadcn/ui or daisyUI.

These components are unique to Plorea. They carry state machine logic, channel awareness, and domain-specific interaction rules.

07 — Policy

Vendor Usage Policy

To avoid UI fragmentation, vendor components must follow strict usage rules. Plorea components must remain the single UI contract across all product surfaces.

✓ Allowed
  • Using vendor components via Plorea wrapper components
  • Direct usage in prototype branches or experiments
  • Direct usage when creating a new canonical Plorea component
  • Using daisyUI classes inside @plorea/components source
✗ Not Allowed
  • Random badge-success classes scattered in product screens
  • Multiple visual implementations of the same UI pattern
  • App-specific theme definitions that override Plorea tokens
  • Importing directly from shadcn/ui in product application code
Principle If you find yourself writing class="badge badge-success" in a product screen, you should be writing <PStatusBadge status="paid" /> instead.
08 — shadcn/ui

When to use shadcn/ui

shadcn/ui is the right tool for building complex, interactive components that require accessible interaction patterns. It handles keyboard navigation, focus management, and ARIA attributes.

Use shadcn/ui for
  • →Complex interactive components
  • →Form elements with validation
  • →Modals and dialogs
  • →Menus and command palettes
  • →Composable UI structures
Typical examples
Dialog DropdownMenu Popover Command Tabs Drawer Form Select Combobox
Remember Always wrap shadcn/ui components in a Plorea component before exposing them to product teams. The Plorea wrapper applies tokens, sets variant constraints, and gives the component a stable public API.
09 — daisyUI

When to use daisyUI

daisyUI is best for simple visual components, quick layout prototyping, and lightweight UI elements that don't require complex interaction logic.

Use daisyUI for
  • →Simple visual components
  • →Fast layout prototyping
  • →Lightweight UI elements
Typical examples
badge alert loading stat card
Wrapper Pattern

Always wrap daisyUI classes inside a Plorea component:

// ✅ Correct — Plorea wrapper <PStatusBadge status="paid" /> // ✗ Not allowed in product screens <span class="badge badge-success"> Paid </span>
10 — UX

POS UX Principles

Plorea POS interfaces follow strict UX principles designed for high-frequency, high-pressure operational environments. These rules are non-negotiable for POS and KDS surfaces.

01
Maximum two clicks for common actions
Any action a cashier performs more than 20 times per shift must be reachable in ≤ 2 taps from the main screen.
02
Primary actions always visible
The primary action on every screen is always above the fold and visually prominent. Never hidden behind a menu.
03
No hidden navigation layers
POS interfaces must not require discovery. All navigation paths are visible. No hamburger menus on operational screens.
04
Visual feedback under 100ms
All interactive elements provide visual feedback within 100ms. Use CSS transitions, not JS animations, for this layer.
05
Color communicates state
State colours are defined in tokens and are never overridden per-component. Green = ready, yellow = in progress, red = error/alert — always.
06
Avoid blocking modal flows
Modals that block the entire POS screen are only acceptable for payment flows and destructive confirmations. Never for informational content.
07
Interface must work without training
A new cashier with restaurant industry experience should be able to process a simple order within 5 minutes on their first day.
11 — Performance

Performance Guidelines

POS terminals are dedicated hardware, often with limited CPU resources. The UI must remain responsive during peak lunch and dinner rushes, with multiple concurrent orders updating in real time.

Render Rules
  • ✗Avoid heavy CSS animations on POS screens
  • ✓Target frame render under 16ms (60fps)
  • ✗Avoid deep component nesting (> 8 levels)
  • ✗Limit unnecessary state updates and re-renders
  • ✓Use skeleton loaders, not spinners
  • ✓Optimistic updates for order state changes
Touch Targets
  • →Minimum: 44×44px (WCAG AA)
  • →POS terminal: 48×48px preferred
  • →Kiosk dine-choice buttons: 120px minimum
  • →Destructive actions: always require confirmation
12 — Accessibility

Accessibility

Accessibility is a first-class requirement, not an afterthought. shadcn/ui provides the accessible interaction primitives — Plorea wrapper components must preserve and extend that foundation.

Requirement All canonical components must pass WCAG AA before they can be promoted from experimental to stable in the component registry.
Requirements
  • ✓Full keyboard navigation
  • ✓Screen reader compatibility (ARIA labels)
  • ✓High contrast colour combinations (4.5:1 min)
  • ✓Visible focus states on all interactive elements
shadcn/ui Advantage

shadcn/ui is built on Radix UI primitives, which handle focus trapping, keyboard navigation, ARIA attributes, and screen reader announcements automatically. Plorea wrapper components inherit this foundation.

Iconography

react-icons is the icon library used across the Plorea platform. It aggregates 40+ icon sets into a single package with ES6 named imports and full tree-shaking — only imported icons appear in the bundle.

Install: npm install react-icons  ·  Docs: react-icons.github.io/react-icons

Preferred icon sets

Primary — Heroicons 2 (hi2)

Outlined, clean stroke. Built by the Tailwind CSS team. Best alignment with Plorea's visual language.

Secondary — Lucide (lu)

Consistent stroke, large catalogue, MIT licensed. Good fallback when hi2 lacks a specific icon.

Usage — React + TypeScript

Basic named import

Typed Icon wrapper component

Centralise sizing and ARIA in one reusable component.

import { type IconType } from 'react-icons'

interface IconProps {
  icon: IconType
  size?: number  // default 20
  className?: string
  label?: string  // aria-label for standalone icons
}

export function Icon({ icon: C, size = 20, className, label }: IconProps) {
  return <C size={size} className={className}
        aria-label={label} aria-hidden={!label} />
}

Implementation — 4 steps

1. Import tokens.css at app root
// main.tsx
import './tokens.css'
import './index.css'
2. Toggle dark class on <html>
// useDarkMode.ts
export function useDarkMode() {
  const toggle = () =>
    document.documentElement
      .classList.toggle('dark')
  return { toggle }
}
3. Respect system preference on load
const saved = localStorage.getItem('plorea-theme')
const osDark = window.matchMedia(
  '(prefers-color-scheme: dark)'
).matches
if (saved === 'dark' || (!saved && osDark))
  document.documentElement.classList.add('dark')
4. Components use tokens — never hardcode
/* ✓ Adapts automatically */
.card { bg: var(--color-bg-surface) }

/* ✗ Breaks in dark mode */
.card { bg: #FFFFFF }
DaisyUI dark theme The plorea-dark DaisyUI theme is defined in tailwind.config.js. Activate with data-theme="plorea-dark" on <html> alongside the dark class — both must be toggled together for full coverage.
Respect system preference Always check prefers-color-scheme: dark on first load before defaulting to light. Restaurant staff often work in dim environments — forcing light mode causes unnecessary eye strain.
13 — Guidance

React Component Token Rules

The token architecture only delivers dark mode (and future theme support) if components consume var(--color-*) tokens exclusively. A single bg-gray-100 in a component bypasses the entire swap mechanism. These are the rules that make the system work end-to-end.

✓ Correct — semantic token
// Card component
<div
  className="rounded-lg p-6"
  style={{
    background: 'var(--color-surface)',
    border: '1px solid var(--color-border)',
    boxShadow: 'var(--elevation-1)'
  }}
>
✗ Wrong — primitive Tailwind class
// Bypasses token swap — stays white in dark mode
<div
  className="bg-white border border-gray-200"
  className="rounded-lg p-6 shadow-sm"
>

// Also wrong — dark: modifier couples component to mode
<div className="bg-white dark:bg-gray-900">

Tailwind classes that are safe to use

Not all Tailwind utility classes break token contract. Layout, spacing, radius and typography scale classes are fine — they carry no colour information.

✓ Safe Tailwind classes
✓ p-6 px-4 py-2 — spacing
✓ gap-3 space-y-2 — layout
✓ rounded-lg rounded-xl — radius
✓ text-xs text-sm text-md — type scale
✓ font-bold font-semibold — weight
✓ flex items-center justify-between
✓ w-full h-11 min-w-0 — sizing
✓ truncate overflow-hidden
✓ transition duration-fast — motion
✓ z-modal z-dropdown — named z-index
✗ Classes that break token contract
✗ bg-white bg-gray-50 — use var(--color-surface)
✗ bg-gray-100 — use var(--color-surface-hover)
✗ text-gray-900 — use var(--color-text-primary)
✗ text-gray-500 — use var(--color-text-muted)
✗ border-gray-200 — use var(--color-border)
✗ shadow-sm shadow-md — use var(--elevation-1)
✗ text-blue-700 — use var(--color-accent)

✗ bg-[#1E1D24] — never hardcode hex

cn() helper — combining tokens with Tailwind

import { clsx, type ClassValue } from 'clsx'
import { twMerge } from 'tailwind-merge'

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}

// Usage — layout classes via cn(), colour via style prop
const Card = ({ className, ...props }) => (
  <div
    className={cn("rounded-lg p-6", className)}
    style={{ background: 'var(--color-surface)',
           border: '1px solid var(--color-border)' }}
    {...props}
  />
)
Pattern: layout in className, colour in style Use cn() for structural Tailwind classes (spacing, radius, flex, sizing). Use inline style prop for all colour, border, and shadow tokens. This keeps className readable while ensuring every colour passes through the token layer.

12 — Usage Rules

Do / Don't

✓ Do
  • One primary CTA per view
  • Always pair buttons: one fill + one outline
  • Use blue-800 for interactive text and links
  • Max 4 input fields per group
  • Use icon + colour for errors (never colour alone)
  • Keep tooltips to 6–8 words
  • Snap all spacing to the 4px grid
  • Use react-icons consistently — pick one set and stick to it
  • Touch targets ≥ 44×44px
  • Confirm destructive actions with a modal dialog
  • Label primary actions with the outcome, not the action type
  • Use data-density to adapt list density per surface
  • Respect elevation levels — cards at 1, dropdowns at 2, modals at 3
  • Use named z-index tokens (z-modal, z-dropdown)
  • Show skeleton states that mirror content shape while loading
  • Include reason + action in every empty state
✗ Don't
  • Use #6BB8DA (primary blue) for body text on white
  • Place two filled or two outline buttons side by side
  • Use gradient fills (solid colours only)
  • Mix outline and filled icon styles
  • Show raw error codes to end users
  • Auto-dismiss success toasts in under 3 seconds
  • Animate width or height directly
  • Let tooltips obscure interactive elements
  • Add more than 5–7 items to a sidebar nav group
  • Hardcode px values outside the spacing scale
  • Skip elevation levels — don't use modal shadow on a card
  • Use btn-primary-hover style classes — use CSS pseudo-states instead
  • Write generic labels: "OK", "Yes", "Submit"
  • Use z-[9999] or other raw z-index integers
  • Show a blank surface while data loads
  • Use a spinner where a skeleton is possible

09c — Patterns

Layout Hierarchy

Four nested levels with fixed spacing rules. Developers should never need to decide what padding a card gets — the level determines it. This eliminates spacing inconsistency across the platform.

Page p-12 (48px) Top-level route container. Full viewport height. Contains sections.
Section gap-8 (32px) Logical grouping of related cards. Separated by 32px gap.
Card p-6 (24px) Primary content container. Elevation-1. White background.
Component gap-3 (12px) Internal component spacing. Buttons, inputs, labels.
Token mapping
/* Layout hierarchy — always use these, never invent new values */
const layoutTokens = {
  page: 'p-12', // 48px — route wrapper
  section: 'gap-8', // 32px — between cards
  card: 'p-6', // 24px — card internal
  component: 'gap-3', // 12px — within components
}
Kiosk / POS exception Touch surfaces use reduced page padding: p-6 (24px) at the page level to maximise usable screen area. Card and component spacing stays the same.

09f — Patterns

Z-Index Scale

Named z-index levels prevent z-[9999] sprawl. Every component that stacks above the document must use one of these values — never a raw integer.

0baseNormal document flow. All static components.
10raisedSticky table headers, floating input labels.
1000dropdownDropdowns, select menus, command palettes, popovers.
1100stickyFixed top nav, fixed sidebar, sticky page headers.
1200overlayDrawer and modal backdrop scrim (surface-overlay).
1300modalModal dialogs, alert dialogs, full-screen kiosk panels.
1400toastToast notifications — must always clear modals.
Tailwind usage
/* ✓ Correct — named token */
<div className="z-modal fixed inset-0">

/* ✗ Wrong — raw integer */
<div className="z-[9999] fixed inset-0">
Elevation + z-index must match An element at elevation-3 (modal) must use z-modal. An element at elevation-2 (dropdown) must use z-dropdown. Mismatching these creates UI where a dropdown appears behind a card.

Conclusion

Summary

The Plorea UI Framework standardizes how user interfaces are built across the entire Plorea platform. By combining modern open foundations with Plorea-specific design tokens, wrapper components, and domain components, the framework enables consistent UX, faster development, scalable architecture, and maintainable frontend systems.

Why this matters Platforms that follow a layered component strategy consistently ship features 2–3× faster after the initial investment, with dramatically lower UI-related bug rates. This is the difference between a mature SaaS platform and a legacy POS system that's hard to change. The Plorea framework is the foundation that makes the rest possible.
