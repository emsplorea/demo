01 — Foundation

Introduction

Plorea is a restaurant SaaS platform covering POS, kiosk, QR and online ordering. The design system spans all touchpoints: back-office dashboard, kiosk surface, operator view, and customer-facing ordering flows.

The design language is clean-operational — precise, fast to scan, and built for high-frequency use by restaurant staff under pressure. Aesthetic influences: Apple HIG (whitespace, type hierarchy), Linear (density, colour use) and Stripe (documentation sharpness).

Tech Stack Vite + React + TypeScript · Tailwind CSS · DaisyUI (plorea theme) · @tailwindcss/typography · Plus Jakarta Sans (body) · Sora (display/headings)

Tailwind class merging (cn utility)

Use clsx + tailwind-merge to safely compose Tailwind classes.

06 — Iconography

Iconography

react-icons is the icon library used across the Plorea platform. It aggregates 40+ icon sets into a single package with ES6 named imports and full tree-shaking — only imported icons appear in the bundle.

Install: npm install react-icons  ·  Docs: react-icons.github.io/react-icons

Preferred icon sets

Primary — Heroicons 2 (hi2)

Outlined, clean stroke. Built by the Tailwind CSS team. Best alignment with Plorea's visual language.

import { HiOutlineHome } from 'react-icons/hi2'
Secondary — Lucide (lu)

Consistent stroke, large catalogue, MIT licensed. Good fallback when hi2 lacks a specific icon.

import { LuReceipt } from 'react-icons/lu'

Usage — React + TypeScript

Basic named import
import { HiOutlineHome, HiOutlineCog6Tooth } from 'react-icons/hi2'

export function NavItem() {
  return (<div className="flex items-center gap-2 text-accent">
    <HiOutlineHome size={20} />
    <span>Dashboard</span>
  </div>)
}
Typed Icon wrapper component

Centralise sizing and ARIA in one reusable component.

import { type IconType } from 'react-icons'

interface IconProps {
  icon: IconType
  size?: number  // default 20
  className?: string
  label?: string  // aria-label for standalone icons
}

export function Icon({ icon: C, size = 20, className, label }: IconProps) {
  return <C size={size} className={className}
        aria-label={label} aria-hidden={!label} />
}
Global defaults via IconContext
import { IconContext } from 'react-icons'

<IconContext.Provider value={{ size: '20', className: 'text-text-secondary' }}>
  <Sidebar />
</IconContext.Provider>

12 — Usage Rules

Do / Don't

✓ Do
  • One primary CTA per view
  • Always pair buttons: one fill + one outline
  • Use blue-800 for interactive text and links
  • Max 4 input fields per group
  • Use icon + colour for errors (never colour alone)
  • Keep tooltips to 6–8 words
  • Snap all spacing to the 4px grid
  • Use react-icons consistently — pick one set and stick to it
  • Touch targets ≥ 44×44px
  • Confirm destructive actions with a modal dialog
  • Label primary actions with the outcome, not the action type
  • Use data-density to adapt list density per surface
  • Respect elevation levels — cards at 1, dropdowns at 2, modals at 3
  • Use named z-index tokens (z-modal, z-dropdown)
  • Show skeleton states that mirror content shape while loading
  • Include reason + action in every empty state
✗ Don't
  • Use #6BB8DA (primary blue) for body text on white
  • Place two filled or two outline buttons side by side
  • Use gradient fills (solid colours only)
  • Mix outline and filled icon styles
  • Show raw error codes to end users
  • Auto-dismiss success toasts in under 3 seconds
  • Animate width or height directly
  • Let tooltips obscure interactive elements
  • Add more than 5–7 items to a sidebar nav group
  • Hardcode px values outside the spacing scale
  • Skip elevation levels — don't use modal shadow on a card
  • Use btn-primary-hover style classes — use CSS pseudo-states instead
  • Write generic labels: "OK", "Yes", "Submit"
  • Use z-[9999] or other raw z-index integers
  • Show a blank surface while data loads
  • Use a spinner where a skeleton is possible
09 — Patterns

Density System

Three density modes let each surface display the right amount of information for its context. Set data-density on any container — all child rows and gaps adapt automatically via CSS custom properties.

Token definitions
/* tailwind.config.js → extend */
density: {
  comfortable: { row: '56px', gap: '16px', py: '14px' },
  compact: { row: '44px', gap: '12px', py: '10px' },
  ultra: { row: '36px', gap: '8px', py: '7px' },
}
When to use each mode
Comfortable Back-office dashboard, settings, onboarding
Compact Order lists, inventory, POS side panels
Ultra KDS, order board, live monitoring, scan lines

Implementation pattern Apply data-density="ultra" to the container element. Child components read var(--row-h), var(--row-gap), var(--cell-py) via CSS. In React, expose a DensityContext so nested components can conditionally render additional detail (e.g. suppress secondary text in ultra mode). Ultra mode is also the basis for scan line layout: single-row, all key data inline, no stacking.

09c — Patterns

Layout Hierarchy

Four nested levels with fixed spacing rules. Developers should never need to decide what padding a card gets — the level determines it. This eliminates spacing inconsistency across the platform.

Page p-12 (48px) Top-level route container. Full viewport height. Contains sections.
Section gap-8 (32px) Logical grouping of related cards. Separated by 32px gap.
Card p-6 (24px) Primary content container. Elevation-1. White background.
Component gap-3 (12px) Internal component spacing. Buttons, inputs, labels.
Token mapping
/* Layout hierarchy — always use these, never invent new values */
const layoutTokens = {
  page: 'p-12', // 48px — route wrapper
  section: 'gap-8', // 32px — between cards
  card: 'p-6', // 24px — card internal
  component: 'gap-3', // 12px — within components
}
Kiosk / POS exception Touch surfaces use reduced page padding: p-6 (24px) at the page level to maximise usable screen area. Card and component spacing stays the same.

Every component follows a predictable three-zone anatomy. New components should match this structure — it makes React implementations consistent and allows developers to infer structure without reading docs.

Content — primary data, body
Footer — metadata, secondary actions
Table

Body — data rows

interface CardProps {
  header?: React.ReactNode
  children: React.ReactNode  // content zone
  footer?: React.ReactNode
  elevation?: 0 | 1 | 2 // default 1
  className?: string
}
TypeScript skeleton — Modal
interface ModalProps {
  title: string
  isOpen: boolean
  onClose: () => void
  children: React.ReactNode
  actions: {
    primary: { label: string; onClick: () => void; isLoading?: boolean }
    secondary?: { label: string; onClick: () => void }
  }
}
Rule Actions in Modals always follow the pattern: primary action right, secondary (cancel/dismiss) left. Primary label must describe the outcome — not just "OK". Example: "Delete order" not "Confirm".
09f — Patterns

Z-Index Scale

Named z-index levels prevent z-[9999] sprawl. Every component that stacks above the document must use one of these values — never a raw integer.

0baseNormal document flow. All static components.
10raisedSticky table headers, floating input labels.
1000dropdownDropdowns, select menus, command palettes, popovers.
1100stickyFixed top nav, fixed sidebar, sticky page headers.
1200overlayDrawer and modal backdrop scrim (surface-overlay).
1300modalModal dialogs, alert dialogs, full-screen kiosk panels.
1400toastToast notifications — must always clear modals.
Tailwind usage
/* ✓ Correct — named token */
<div className="z-modal fixed inset-0">

/* ✗ Wrong — raw integer */
<div className="z-[9999] fixed inset-0">
Elevation + z-index must match An element at elevation-3 (modal) must use z-modal. An element at elevation-2 (dropdown) must use z-dropdown. Mismatching these creates UI where a dropdown appears behind a card.

09h — Patterns

Loading & Skeleton States

Three loading patterns for different contexts. Always show a loading state — never leave a surface blank while data fetches. Prefer skeletons over spinners for content areas; reserve spinners for single-action feedback.

Skeleton — content areas

Use when loading a list or card. Mirror the shape of real content.

Skeleton — card

Use when loading a single card or form.

Inline spinner — action feedback

Use inside a button or inline after a mutation. Not for page-level loading.

Empty state

After loading completes with no results. Always include a reason and an action.

No orders yet today
Orders will appear here as they come in.

    <div className="animate-skeleton h-3 w-14 ml-auto rounded-full bg-gray-200" />

Rules Skeleton shapes must mirror real content — same width, height and grid position. Never show a generic spinner where a skeleton is possible. Empty states must always include: icon + title + reason + action. Loading buttons must disable pointer-events and show a spinner icon, never just change label text.
