Retry Policy + Delivery Guarantees
At-least-once delivery — duplicates possible, receiver must dedup on eventId
Retry: exponential backoff — 1s, 5s, 30s, 5min, 30min (5 attempts max)
Dead-letter: after max retries → DLQ + ops alert → manual review
Expected response: 200 OK within 5s timeout

Retry Initiated
SymptomOrder stays in "Pushed" state — not Acknowledged Cause3rd party webhook endpoint exceeded 5s timeout ActionAuto-retry with backoff (5 attempts). After max: DLQ + ops alert Owner3rd Party (must respond within 5s)
No Retry (Client Error)
SymptomOrder push immediately fails — goes to DLQ Cause3rd party returned 4xx — permanent rejection (bad payload, auth, etc.) ActionManual investigation. Fix 3rd party endpoint, then replay from DLQ Owner3rd Party (must fix webhook endpoint)
Idempotent — Safe
Symptom3rd party receives same eventId twice — no side effects CausePlorea retried after timeout, but 3rd party had already processed Action3rd party returns 200 OK — Plorea marks as Acknowledged Owner3rd Party (must dedup on eventId)

Governance Rules
Architecture Governance
No duplicated logic — availability computed once, used everywhere
No UI business logic — channels are presentation only
Extend don't fork — new channels reuse Order Core, never clone
Integration boundary — gate enforced, financial surfaces blocked
Deterministic availability — same input → same output, always
Role matrix mapping — external roles never trusted 1:1
At-least-once delivery — receiver must dedup on eventId
No POS bypass — all orders go through Order Core state machine
🚫 What We Refuse To Do
✕ No financial reporting in Plorea (integration mode)
✕ No channel-specific order logic — unified engine only
✕ No UI-based business logic — channels are presentation layer
✕ No settlement or Z-rapport in Plorea — stays in 3rd party
✕ No direct role passthrough — always mapped via matrix
